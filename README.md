# 断裂学习法 · Fracture Learning

> **真知产生于断裂点的精准修复。**

一个基于苏格拉底式提问的 Agent Skill。不给答案，先问思路，找到你的认知断裂点，修复它，提炼成可复用的决策规则。

---

## 核心理念

传统教学：老师讲 → 学生听 → 考试忘

断裂学习法：**问你的思路 → 找到你卡在哪 → 修复断裂点 → 提炼规则**

你不会得到直接答案。你会得到：
- 一个帮你理清思路的提问者
- 一个精准定位你错误的诊断者
- 一条"看到 X，就用 Y"的决策规则

---

## 工作流程

```
你提问
  │
  ├─ 你说了思路？ ──否──→ 先问你思路
  │       │
  │      是
  │       ↓
  ├─ 思路对吗？
  │   ├─ 对 → 追问变式，检验理解深度
  │   ├─ 错 → 指出具体第几步错，追问为什么
  │   └─ 缺 → 指出跳过了哪步，追问这一步
  │
  ├─ 🔴 确认卡点："你的卡点是 X，对吗？"
  │
  ├─ 三步修复：
  │   1. 定位：你把 A 和 B 混淆了
  │   2. 追根：你为什么这样想？
  │   3. 提炼：看到条件，就用方法
  │
  ├─ 🔴 确认理解："你能用自己的话说说吗？"
  │
  ├─ 📝 输出决策规则
  │
  └─ 追问变式，验证掌握
```

## 适用场景

| 你在说 | 这个 skill 会帮到你 |
|--------|-------------------|
| "这题怎么做？" | 不给答案，先问你思路 |
| "我不理解为什么…" | 找到你认知断裂的具体位置 |
| "帮我讲解一下…" | 用提问引导你自己想通 |
| "教我…" | 通过对话建立深层理解 |

## 不适用场景

| 你在说 | 建议 |
|--------|------|
| "帮我查一下 X 的用法" | 这是查资料，不是学习，不需要这个 skill |
| "帮我写一段代码" | 这是生成任务，不是理解任务 |
| "总结一下这篇文章" | 这是信息提取，不是断裂修复 |

## 效果示例

**你问：** "为什么递归会导致栈溢出？"

**Agent 回复：**
> 你的思路是什么？先说说你怎么想的——
> 1. 你对"调用函数时内存里发生了什么"有概念吗？
> 2. 你觉得递归和普通函数调用，在内存占用上有什么区别？
> 3. 你从哪一步开始不确定？

**你答：** "我觉得每次递归都会创建新变量，内存不够就溢出了"

**Agent 回复：**
> 你说的"每次递归创建新变量"方向对，但不够精确。你的卡点是：你没区分"变量"和"栈帧"。
> 你为什么认为是"变量"导致的内存不够？

**你答：** "因为函数里有局部变量啊"

**Agent 回复：**
> 对，但关键不只是变量。每次函数调用，系统会创建一个**栈帧**，里面包含：局部变量、参数、返回地址。
> 你能用自己的话说说，为什么递归比循环更容易栈溢出吗？

**你答：** "因为递归每次调用不释放栈帧，循环没有这个问题"

**Agent 回复：**
> 📝 决策规则：看到递归调用，就想"栈帧会累积"，因为每次调用不释放，直到返回才弹出。
>
> 那如果是一个尾递归优化的语言呢？这个规则还成立吗？

## 设计依据

- **苏格拉底提问法**：通过提问引导思考，而非直接传授
- **认知断裂理论**：学习障碍来自认知链的断裂点，修复断裂比灌输知识更有效
- **决策规则提取**：将理解固化为"看到 X，就用 Y"的形式，便于迁移应用
